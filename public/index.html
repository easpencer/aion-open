What Aion Does

Four pillars

1

Consolidate

Pull health data from every source — Apple HealthKit, Google Health Connect, wearables, clinical records — into a single encrypted FHIR R4 store on your phone.

2

Understand

On-device AI models explore your data — medications, vitals, labs, conditions — and answer health questions without anything leaving your device.

3

Act

Agentic AI with tool calling. Search the web, set reminders, look up drug interactions, share content — with tiered consent at every step.

4

Bridge

When you want more compute, bridge your data to your PC over WiFi or USB. Run Ollama, Claude Code, Python scripts — still your network, still your data.

Ways to Use the Bridge

Four paths, all build on the same API

The bridge exposes your health data as a standard API. How you consume it is up to you.

1. As a raw API

Use the REST endpoints from any language, tool, or app you build. The bridge is just an HTTP server — curl, requests, fetch(), Postman, anything.

curl http://localhost:3000/api/Observation?category=vital-signs
curl http://localhost:3000/api/Condition
curl -X POST http://localhost:3000/api/analyze \
  -d '{"question":"BP trends?"}'

2. With local AI models

Run Ollama on your PC. The bridge’s /api/analyze endpoint routes questions to your local LLM. Your data + your model = fully private analysis.

# Install Ollama + pull a model
ollama pull llama4:scout

# The bridge auto-detects Ollama
# Ask questions via API or console
curl -X POST localhost:3000/api/analyze \
  -d '{"question":"Summarize my medications"}'

3. From the terminal

The CLI tool pipes data to jq, scripts, cron jobs, or any unix workflow. Great for automation and monitoring.

# Query and pipe to jq
./cli.js query Observation --category vital-signs \
  | jq '.[] | .code.text'

# Export for backup
./cli.js export --format bundle > health.json

# Analyze
./cli.js analyze "Any drug interactions?"

4. Via the web console

Open http://localhost:3000 for a visual browser. Query resources, inspect JSON, generate code snippets, test API calls — no coding required.

The console is also where you connect to your phone’s bridge if you didn’t set env vars at startup. Enter the URL and pairing code, and you’re live.

All four paths talk to the same REST API. Build a dashboard, a Jupyter notebook, a Claude Code integration, a cron monitor — or all of them at once.

PC Models

Recommended models for Ollama

When bridging to your PC, these models work well with the analyze endpoint. Aion auto-detects installed models.

Best for Most PCs

Llama 4 Scout — 17B active (MoE, 16 experts). Multimodal. 10M context. ~12GB VRAM. ollama pull llama4:scout Qwen 3 8B — Best 8B for tool calling. ~5GB. ollama pull qwen3:8b Gemma 3 4B — Runs on CPU. No GPU needed. ollama pull gemma3:4b

High-End (20GB+ VRAM)

Llama 4 Maverick — 17B active (MoE, 128 experts). Top-tier multimodal. ~24GB VRAM. ollama pull llama4:maverick Qwen 3 32B — Excellent tool calling + health reasoning. ~20GB. ollama pull qwen3:32b Phi-4 14B — Strong reasoning model from Microsoft. ~8GB. ollama pull phi4:14b

Install any model: ollama pull model-name — the bridge settings UI on your phone auto-detects available models when connected.

Get Started

Four steps to your data

1

Enable bridge on your phone

Open Aion → Settings → AI & Developer tab → scroll to Local Bridge → Toggle ON. Note the 6-digit pairing code and connection URL (e.g. wss://192.168.1.42:8420). Choose WiFi or USB mode.

2

Start the server

git clone https://github.com/easpencer/aion-open.git
cd aion-open && npm install
BRIDGE_URL=wss://192.168.1.42:8420 PAIRING_CODE=123456 npm start
3

Query your data

# Vital signs
curl http://localhost:3000/api/Observation?category=vital-signs

# Conditions
curl http://localhost:3000/api/Condition

# Ask a question (requires Ollama running)
curl -X POST http://localhost:3000/api/analyze \
  -H "Content-Type: application/json" \
  -d '{"question":"What are my BP trends?"}'
4

Or use the CLI (in another terminal)

# Requires the server from step 2 running in another terminal
./cli.js query Observation --category vital-signs --limit 20
./cli.js get Observation obs-hr-001
./cli.js analyze "Should I be concerned about my blood pressure?"
./cli.js export --format bundle > health.json

USB Mode (no WiFi needed)

Connect your phone via USB cable and forward the port. Faster than WiFi, works without any network.

# iOS (requires libimobiledevice: brew install libimobiledevice)
iproxy 8420 8420

# Android (requires adb from Android SDK platform-tools)
adb forward tcp:8420 tcp:8420

# Then use wss://localhost:8420 as BRIDGE_URL
REST API

Endpoints

All endpoints return JSON. Connect first (via env vars or POST /api/connect), then query freely.

METHOD ENDPOINT DESCRIPTION
GET/api/{type}Query FHIR resources. Params: category, limit, offset, start, end, sort
GET/api/{type}/{id}Get a single resource by ID. Returns 404 if not found.
POST/api/analyzeAsk a health question. Body: {"question":"..."}. Uses local LLM (Ollama).
GET/api/_exportExport as FHIR Bundle or CSV. Params: format, resourceType
GET/api/_metadataFHIR R4 CapabilityStatement. Lists supported resource types and interactions.
GET/api/_auditView bridge audit log. Params: limit
POST/api/connectConnect to phone. Body: {"url":"wss://...","code":"123456"}
GET/api/statusCheck connection status. Returns connected, bridgeUrl, error.
Code Examples

Build with any language

The REST API works with any HTTP client. No SDK needed.

Python
import requests

# Query vital signs
resp = requests.get(
    "http://localhost:3000/api/Observation",
    params={"category": "vital-signs", "limit": 20}
)
data = resp.json()

for entry in data.get("entry", []):
    r = entry["resource"]
    name = r.get("code", {}).get("text", "?")
    val = r.get("valueQuantity", {})
    print(f"  {name}: {val.get('value')} {val.get('unit', '')}")

# Ask a health question
resp = requests.post(
    "http://localhost:3000/api/analyze",
    json={"question": "What should I discuss at my next visit?"}
)
print(resp.json().get("answer", ""))
JavaScript / Node.js
// Query vital signs
const resp = await fetch(
  "http://localhost:3000/api/Observation?category=vital-signs&limit=20"
);
const data = await resp.json();

for (const { resource } of data.entry || []) {
  const name = resource.code?.text;
  const val = resource.valueQuantity;
  console.log(`  ${name}: ${val?.value} ${val?.unit || ""}`);
}

// Export all data as FHIR Bundle
const exp = await fetch("http://localhost:3000/api/_export");
const { data: bundle, total } = await exp.json();
console.log(`Exported ${total} resources`);
Claude Code Integration

Talk to your health data in Claude Code

Add the MCP server to Claude Code. Then ask health questions in natural language — Claude queries your real FHIR data.

1. Build the MCP server

cd aion-web/src/services/bridge/mcp
npm install && npm run build

2. Add to Claude Code settings

{
  "mcpServers": {
    "aion-health": {
      "command": "node",
      "args": ["path/to/mcp/dist/index.js"],
      "env": {
        "AION_BRIDGE_URL": "wss://192.168.1.42:8420",
        "AION_PAIRING_CODE": "123456"
      }
    }
  }
}

Security

Private by design

Data never leaves your local network. Multiple layers of protection.

TLS encryption (self-signed cert, SHA-256 fingerprint in app)
6-digit pairing code (rate-limited: 5 attempts, 60s cooldown)
HIPAA Safe Harbor de-identification on by default
Audit log of all queries (clientId, type, timestamp)
LAN-only — never exposed to internet
USB mode for zero-network operation
Ephemeral auth — tokens expire when bridge stops
De-id fails closed — never serves raw data on error
